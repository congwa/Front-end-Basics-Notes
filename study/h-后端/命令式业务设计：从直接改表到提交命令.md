# 命令式业务设计：从直接改表到提交命令

## 写在前面

这是一篇讲“为什么要用命令式设计”的文档，而不是讲数据库语法或代码框架。

读完应该能回答三个问题：

- 为什么有些业务直接 update 一张表就行，有些却要绕一大圈？
- 命令式架构到底解决了什么问题？
- 我自己的项目什么时候该上、什么时候不该上？

下面用一个虚构业务一步步演化，不涉及具体技术栈。

---

## 一、起点：一个最朴素的业务

假设我们做了一个「能力订阅」系统。用户可以订阅某项能力，订阅之后能用，不订阅就不能用。

数据库里只有一张表：

```
subscription
  - account_id      谁
  - capability_key  订阅了什么
  - is_enabled      是否启用
  - config          配置
  - status          状态：active / cancelled
```

最初产品需求只有两个：

- 用户点「订阅」 → 写一条 `active` 的记录
- 用户点「取消」 → 把 `status` 改成 `cancelled`

工程师很自然写出：

```text
INSERT INTO subscription(...) VALUES(...);
UPDATE subscription SET status='cancelled' WHERE ...;
```

这时候一切都好。

---

## 二、第一个问题：业务规则散得到处都是

一个月后，业务方加了若干规则：

- 没有这个项目权限的人不能订阅
- 已经订阅过的人不能再订阅
- 试用期内不能取消
- 某些能力下架了，不能再被订阅
- 付费能力不能直接订阅，要先走付款

每加一条规则，前端就要在调用 update 前补一段判断。
更糟的是，订阅入口不止一个：

- 能力市场页面
- 能力详情页
- 设置页里的「快速开通」
- 运维后台手动开通
- 未来支付回调自动开通

**结果是同一份业务规则被复制了五份。**

只要有一个地方漏了校验，数据就会出问题。
而最容易漏的，恰恰是新加的规则。

> 第一个教训：业务规则不能散在调用方，必须收口到一个入口。

### 第一步解决：抽出一个「业务函数」

不管谁要改 `subscription`，都必须调用同一个函数：

```text
subscribe(account_id, capability_key, plan)
cancel(account_id, capability_key)
update_config(account_id, capability_key, config)
```

所有校验、所有写入都在这些函数里完成，外部只能调用，不能直接改表。

到这一步，已经比第一阶段进步很多。但还会遇到下一个问题。

---

## 三、第二个问题：重复请求

线上突然出现一些奇怪的现象：

- 同一个用户在几秒内被扣了两次费
- 一个能力出现两条订阅记录
- 用户截图说自己点了一次，但是后台日志里有三次请求

原因可能是：

- 用户手抖连点
- 网络慢，前端自动重试
- 用户刷新页面，浏览器把表单又交了一次
- 服务端处理成功了，但响应没回到前端，前端自以为失败重试

普通 update 是没办法防这种情况的。
即便加了「按钮 loading 状态」，也只能防一部分。

> 第二个教训：写入类操作必须有「这是不是重复请求」的判断能力。

### 第二步解决：加幂等键 idempotency_key

让客户端每次发起业务动作时，带一个唯一编号，比如 UUID。

服务端的逻辑变成：

```text
1. 先查这个 idempotency_key 有没有处理过
   - 处理过 → 直接返回上次的结果
   - 没处理过 → 继续往下走
2. 执行业务
3. 把结果存下来，下次同样 key 来了能命中
```

这就要求服务端有一张表能保存「上次的处理结果」。

到这里，单纯一张当前状态表 `subscription` 已经不够了。
必须再加一张表，专门记每一次「业务动作」。

---

## 四、第三个问题：用户说没生效，你查不出原因

用户反馈：「我刚刚明明开通了，但是页面没生效。」

如果只有一张状态表，工程师能查到的就是：

- 当前是什么状态

查不到：

- 用户到底点没点过？
- 请求收到了吗？
- 校验到哪一步失败的？
- 是权限不够，还是能力下架了，还是数据库失败？
- 服务器返回了什么？

如果团队还有「操作日志」，能稍微查到一点点：

- 用户某时点击了某按钮

但「点过」和「业务有没有完整执行成功」是两回事。

> 第三个教训：写入类业务需要保留「过程」，而不只是「结果」。

### 第三步解决：把动作建成一个状态机

把每一次业务动作都建成一行记录，带上状态字段：

```
command_log
  - id
  - idempotency_key
  - action               订阅 / 取消 / 改配置
  - status               received / validating / executing / succeeded / rejected / failed
  - error_code           失败时的错误码
  - retryable            是否建议重试
  - request_payload      原始请求
  - response_payload     最终结果
  - created_at / updated_at
```

执行流程是：

```text
1. 收到请求 → 先写一行 received
2. 校验参数 / 权限 → 改 validating
3. 真正执行 → 改 executing
4. 写完结果 → succeeded 或 rejected / failed
```

这张表既是日志，也是幂等凭证，还是状态机。

到这一步，命令式雏形已经出来了。

---

## 五、第四个问题：状态变化没有「为什么」

业务越走越深，又出现新问题：

- 一个订阅从 `active` 变成 `cancelled`，是用户取消的，还是过期自动取消的？
- 是付款失败回退的，还是运维手动取消的？
- 上一次状态是什么？变化前后启用状态是怎样的？

只有当前状态表，回答不了。

如果业务涉及钱、权益、对账，这些问题迟早要回答。

> 第四个教训：状态本身不解释自己，必须有「状态变化流水」。

### 第四步解决：加一张权益变化流水表

```
entitlement_event
  - id
  - source_type       来源：command / payment / manual / reconcile
  - source_id         来源 id（指向 command_log / order / 操作员）
  - capability_key
  - old_status / new_status
  - old_enabled / new_enabled
  - old_expires_at / new_expires_at
  - reason            人能读懂的原因
  - metadata          额外信息
  - created_at
```

状态表 `subscription` 只保存「现在」。
流水表 `entitlement_event` 保存「过去发生过什么」。

排查问题、对账、审计，都靠这张表。

---

## 六、第五个问题：以后要接支付

现在所有能力都是免费的，但产品已经在规划付费版本。

如果将来直接在 `subscription` 上加付费逻辑，会非常痛：

- 支付涉及订单、退款、对账，业务字段一下子翻倍
- 未支付完成时，权益不能立刻给
- 第三方支付平台会重复发回调，必须去重
- 退款后权益要被精确回滚

> 第五个教训：商业化字段不要塞进权益表，应该独立一层。

### 第五步解决：加订单表 + 支付事件表

```
order
  - id
  - account_id / capability_key / plan_key
  - source           free / manual / payment
  - status           created / payment_pending / paid / refunded / ...
  - amount / currency
  - provider / session_id
  - idempotency_key

payment_event
  - id
  - provider / event_id      第三方平台的事件号
  - order_id
  - status                   received / processed / duplicate / failed
  - raw_payload
  - UNIQUE(provider, event_id)   支付平台重复回调时去重用
```

现在虽然没接支付，但表先建好。
将来接入时不需要重构权益层，只是多写两张表里的记录。

## 七、最终架构

到这里，系统从「一张表」演化成了「一组表 + 一个命令入口」。

它的样子是：

```
            ┌──────────────────────────────┐
   外部 →   │  统一命令入口（execute_command） │
            └──────────────┬───────────────┘
                           │
        ┌──────────────────┼─────────────────────────────┐
        ▼                  ▼                             ▼
   command_log      entitlement_event           subscription / order / payment_event
   （命令账本）       （状态变化流水）              （当前状态 + 商业凭证）
```

各表分工：

| 表                   | 职责                                                  |
| -------------------- | ----------------------------------------------------- |
| `subscription`       | 当前权益状态，读多写少                                |
| `command_log`        | 每一次命令的状态机：受理、校验、执行、成败、幂等     |
| `entitlement_event`  | 权益变化流水：从什么状态变成什么状态、为什么、谁导致 |
| `order`              | 商业凭证：免费授权 / 人工开通 / 付款订单            |
| `payment_event`      | 支付平台原始回调，去重、可重放                        |

外部看到的只有一个入口：

```text
execute_command(action, account_id, capability_key, payload, idempotency_key)
```

不管谁调用，规则只有一份。

---

## 八、它到底解决了什么

汇总一下五步演化解决的事：

1. **业务规则收口**：所有写入必须经过一个入口，规则不再散落。
2. **幂等保护**：重复请求只生效一次。
3. **过程可追溯**：能查到每条命令卡在哪一步、为什么失败。
4. **状态变化可解释**：不是「现在是 X」，而是「从 Y 变成 X，因为 Z」。
5. **商业能力可扩展**：免费 / 人工 / 支付走同一套抽象。

一句话总结：

> 不要让外部直接改最终状态，而是让外部提交一个业务意图，由系统统一校验、执行、记录结果。

## 九、和「操作日志」的区别

很多团队都已经有一张通用「操作日志」表，记录用户做了什么。
那命令表岂不是重复？

差别在「定位」：

- **操作日志**：面向人，回答「谁在什么时候做了什么」，用于审计、行为分析、贡献统计。一般写入即丢手，不影响业务。
- **命令表**：面向系统，回答「这条业务请求被接受了吗、执行到哪里了、要不要重试」。它本身参与业务执行，是状态机的一部分。

举个对比：

用户点「开通能力」。

操作日志会写：

```
张三 在 12:00 点击了「开通能力 X」
```

命令表会写：

```
command_id = abc-123
action = subscribe
status = succeeded
idempotency_key = uuid-xxx
request_payload = {...}
response_payload = {...}
duration_ms = 187
```

权益流水会写：

```
能力 X 从 inactive 变为 active，原因：用户主动订阅，来源：command/abc-123
```

三张表方向不同：操作日志是「人的行为」，命令表是「系统接受请求的过程」，权益流水是「业务事实」。

实际项目里，操作日志可以选择性记录命令的少量字段（比如 action、result），用作前台展示。**命令表不暴露给普通用户，只给后端排查用。**

## 十、什么时候值得用

适合的场景：

- 涉及钱：支付、订阅、退款、积分。
- 涉及权益：开通、关停、过期。
- 涉及多状态流转：待审核 → 通过 → 生效 → 过期。
- 涉及外部系统：支付平台、第三方回调，必然有重复请求。
- 涉及多人协作：运维改一次、用户改一次、自动任务改一次，必须能区分。
- 涉及合规审计：监管或客户要求事后能追溯。

不适合的场景：

- 改个昵称、保存一个 UI 偏好。
- 没有审计、没有重试、没有钱、没有状态流转的纯展示后台。
- 一次性脚本、内部工具。

判断标准很简单：**有没有「客服来问『为什么我刚才那次操作没生效』」的可能？** 有 → 上命令式。没有 → 不用。

## 十一、落地时常见的坑

1. **不要把 `command_log` 当业务主表**
   主表永远是当前状态表（如 `subscription`），命令表只是过程。读取业务时不要 join 命令表。

2. **`idempotency_key` 必须由客户端生成**
   不能服务端生成，否则失去意义。客户端要保证「同一个业务意图」用同一个 key，重试时不要换。

3. **命令表会快速膨胀，要规划清理策略**
   按月归档，或者按 `status = succeeded` 保留 N 天。失败和异常的多保留一些。

4. **校验失败也要落库**
   即便参数错、权限不够，也要写一行 `rejected` 状态的命令。否则线上排查时只能看到「没记录」，不知道是没请求还是被拒绝了。

5. **状态机要画图**
   `received → validating → executing → succeeded / rejected / failed / retrying`，团队所有人对状态流转的理解必须一致，否则很快就会乱。

6. **不要在命令里嵌套发起新命令**
   一条命令只做一件事。如果一个动作要触发其他动作，让上层编排，不要互相调。否则幂等、状态、错误传播会变得难以维护。

7. **错误码要稳定**
   `error_code` 是给前端和未来重试机制看的，必须是稳定的字符串常量，不能是动态拼接的描述。

8. **权益流水和命令表不要二合一**
   它们职责不同：命令表是「请求过程」，流水表是「业务事实」。一个失败的命令不应该写流水，但要留命令记录。

9. **不要在命令表里存大字段**
   `request_payload` / `response_payload` 控制大小，必要时只存摘要或外链 ID，否则表会变得非常臃肿。

10. **当前状态表的写权限要收紧**
    既然规定了所有写入走命令入口，就要在数据库层面禁止外部直接写状态表。否则只是「自觉走流程」，迟早有人绕过。

## 十二、用一个生活类比收尾

可以把这套架构想成餐厅：

- **状态表**：厨房库存。任何时候你只想知道「现在还有多少食材」。
- **命令表**：点单系统。每张订单都有「下单 → 已接受 → 制作中 → 已上菜 / 已退单」。
- **权益流水**：库存变化账本。某食材从 5 份变成 3 份，因为 12 号桌点了两份。
- **订单表**：账单。这桌付了多少钱、用什么方式付的。
- **支付事件**：支付平台原始回单。哪怕银行重复回了三次，餐厅只入账一次。

外部顾客看不到库存，他们只能「下单」。
厨房不接待顾客，只听从订单系统。
吃完结账，账单和库存对得上，钱和回单对得上。

这就是命令式业务设计。

## 十三、回到最初的问题

如果一开始你只是想让用户「订阅一下能力」，写两条 SQL 就够了。

但只要业务往前走一步——加规则、加重试、加排查、加付费——你就会发现，每一步加的东西，命令式架构都已经替你想好对应的位置：

- 规则 → 命令入口
- 重试 → idempotency_key
- 排查 → 命令状态 + 错误码
- 解释状态 → 权益流水
- 付费 → 订单 + 支付事件

不是这个架构复杂，而是业务一旦走到这里，复杂是躲不掉的。
区别在于：**复杂会被收纳成清晰的层，还是散在系统每个角落。**

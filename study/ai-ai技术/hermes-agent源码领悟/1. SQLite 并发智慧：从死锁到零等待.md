# SQLite 并发智慧：从死锁到零等待：SQLite 高并发的两条黄金法则（BEGIN IMMEDIATE + CAS）


1. 多 writer 写同一个 db：用 `BEGIN IMMEDIATE` + 短 timeout + 应用层带 jitter 的重试，别去调 `busy_timeout`。
2. 多 worker 抢同一批任务：用 `UPDATE ... WHERE 期望值` 做 CAS，输家不重试，直接换目标。


## 第一部分：`BEGIN IMMEDIATE` + 短 timeout + jitter 重试

### SQLite 的锁

SQLite 是文件数据库，没有常驻 server。多个进程/线程开同一个 `.db` 文件，靠操作系统文件锁互相协调。

锁状态简化版：

```
UNLOCKED → SHARED (读锁，可共享) → RESERVED (打算写) → EXCLUSIVE (正在写)
```

规则就两条：

- 同一时刻可以有多个 reader（SHARED 锁）。
- 同一时刻只能有一个 writer（EXCLUSIVE 锁）。

WAL 模式改进了一点：reader 不再阻塞 writer，但 writer 之间还是互斥。

### 三种事务模式

```sql
BEGIN;            -- 等同于 BEGIN DEFERRED（默认）
BEGIN IMMEDIATE;  -- 立刻申请写锁
BEGIN EXCLUSIVE;  -- 立刻申请互斥锁
```

`BEGIN DEFERRED`（默认）的行为：开始事务时不申请任何锁，执行第一条 SELECT 才申请 SHARED 读锁，执行第一条 INSERT/UPDATE/DELETE 才升级为 RESERVED 写锁。

`BEGIN IMMEDIATE` 的行为：开始事务就直接申请 RESERVED 写锁，拿不到立刻报 `SQLITE_BUSY`。

差别看起来不大，但下一步就要命了。

### DEFERRED 的死锁陷阱

两个进程 A 和 B 同时干同一件事：先读一个值，根据值再写。

```python
BEGIN              # DEFERRED，不申请锁
SELECT balance ... # 拿到 SHARED 读锁
# ... 业务计算 ...
UPDATE balance ... # 想升级到写锁
COMMIT
```

时间线：

```
t=0    A: BEGIN                  → 无锁
t=0    B: BEGIN                  → 无锁
t=1    A: SELECT                 → A 拿 SHARED
t=1    B: SELECT                 → B 也拿 SHARED（共享，OK）
t=2    A: UPDATE → 升级 RESERVED → 失败：B 还持有 SHARED → SQLITE_BUSY
t=2    B: UPDATE → 升级 RESERVED → 失败：A 还持有 SHARED → SQLITE_BUSY
```

两个都卡死。而且这种 BUSY 不能简单 retry——A 已经读到的 balance 值，可能马上要被 B 改掉，A 如果「睡一会儿再 UPDATE」是基于陈旧值在改，逻辑错误。SQLite 干脆让你回滚整个事务。

这就是写锁升级死锁。

### IMMEDIATE 怎么解决

```python
BEGIN IMMEDIATE    # 一开始就抢写锁
SELECT balance ...
UPDATE balance ...
COMMIT
```

时间线：

```
t=0    A: BEGIN IMMEDIATE → A 拿到 RESERVED
t=0    B: BEGIN IMMEDIATE → 失败 → SQLITE_BUSY
t=1    B 收到 BUSY，没读过数据，sleep 一下重试
t=2    A: SELECT ... UPDATE ... COMMIT → 释放
t=3    B: BEGIN IMMEDIATE → 成功
```

冲突点提前到事务开始的瞬间。B 还没读任何数据、没建立任何假设，retry 完全安全。

DEFERRED 是读完再争写锁，争不过会死；IMMEDIATE 是开局就争写锁，争不过没读过数据，可以随便重试。

### `timeout` 是什么，为什么要短

```python
sqlite3.connect("db.sqlite", timeout=5.0)
```

这个 `timeout=5.0` 调的是 SQLite 的 busy handler：遇到 SQLITE_BUSY 时 SQLite 自己在 C 层内部 sleep + 重试，最多撑到 5 秒，超时才把错误抛给你。

听起来挺贴心，但它有三个坑：

第一，sleep 是确定性的。SQLite 内置 busy handler 按固定时间表 sleep（1ms, 2ms, 5ms, 10ms, 15ms, 20ms, 25ms, 25ms ...），所有撞锁的连接看到的时间表都一样。

第二，由此产生 convoy effect（车队效应）：

```
t=0    赢家 W 在写，A、B、C 都 BUSY
t=1ms  A、B、C 同时被唤醒，同时去抢 → 只有一个赢
t=3ms  又同时被唤醒，又同时撞
...
```

跟高速路堵车一样，前面一辆车走了，后面所有车一起挪一步又撞上。

第三，你的代码看起来在干活，其实在 C 层默默 sleep 了 30 秒。从外面看像 TUI 卡死，毫无可观测性。

### 把重试搬到应用层 + 加 jitter

```python
timeout = 1.0   # SQLite 内部最多等 1 秒
MAX_RETRIES = 15
for attempt in range(MAX_RETRIES):
    try:
        conn.execute("BEGIN IMMEDIATE")
        ...
        conn.commit()
        break
    except sqlite3.OperationalError as e:
        if "locked" in str(e) or "busy" in str(e):
            time.sleep(random.uniform(0.020, 0.150))   # 随机 20-150ms
            continue
        raise
```

jitter 解决 convoy 的原理：

```
t=0    赢家 W 在写，A、B、C 都 BUSY
       A 随机 sleep 27ms
       B 随机 sleep 89ms
       C 随机 sleep 134ms
t=27ms  A 醒来，W 还没写完 → 又 BUSY → 再 sleep 53ms（新随机值）
t=80ms  A 醒来，W 走了 → A 赢
t=89ms  B 醒来 → A 在写 → 再 sleep 60ms
...
```

随机让竞争者错开，不再同时撞墙。

短 timeout 的好处：

- 失败快、能日志、能 metrics、能告警，30 秒在 C 层 sleep 你毫无办法。
- sleep 期间主线程可以做别的事（释放进程内的 Python 锁，渲染 UI）。
- 重试策略（jitter 范围、最大次数）全在你手里。

### Hermes 的真实代码

```python
self._conn = sqlite3.connect(
    str(self.db_path),
    timeout=1.0,
    isolation_level=None,   # 关键：关掉 Python 的隐式 BEGIN
)

for attempt in range(15):
    try:
        with self._lock:
            self._conn.execute("BEGIN IMMEDIATE")
            fn(self._conn)
            self._conn.commit()
        return result
    except sqlite3.OperationalError as exc:
        if "locked" in str(exc) or "busy" in str(exc):
            time.sleep(random.uniform(0.020, 0.150))
            continue
        raise
```

顺带一句 `isolation_level=None` 是必须的。Python sqlite3 模块默认会偷偷在执行 DML 时帮你 `BEGIN`，这会和手写的 `BEGIN IMMEDIATE` 打架报嵌套事务错。设成 None 就是 autocommit 模式，事务边界全由代码控制。

## 第二部分：跨进程协调用 CAS

### 场景

任务表：

```
tasks:
| id  | status | worker |
| t1  | ready  | NULL   |
| t2  | ready  | NULL   |
```

5 个 worker 进程同时抢：找一个 `status='ready'` 的，把它改成 `running`、worker 字段写自己的名字，然后去干活。

需求：同一个任务绝不能被两个 worker 同时领走。

### 朴素「加锁」思路的问题

```python
acquire_lock("tasks_table")
task = SELECT * FROM tasks WHERE status='ready' LIMIT 1
UPDATE tasks SET status='running', worker='me' WHERE id=task.id
release_lock("tasks_table")
```

四个问题：

1. 这把锁本身得有地方实现——Redis、ZooKeeper、文件锁？又多一个组件。
2. 所有 worker 在锁前排队，5 个并发退化成单线程。
3. 持锁的 worker 崩了就死锁，得加 TTL、心跳、续约、看门狗。
4. 用 SQLite 自己的 `BEGIN EXCLUSIVE` 当锁也一样，写操作全是串行的。

### CAS

CAS 来自底层 CPU 指令，意思是「我期望这个变量当前是 X，请把它原子地改成 Y；如果当前不是 X，就别改，告诉我」。

放到 SQL 里：

```sql
UPDATE tasks
SET status = 'running', worker = 'me'
WHERE id = 't1'
  AND status = 'ready'   -- ← 期望值
```

单条 UPDATE 在 SQL 里是原子的。SQLite 用写锁把这些 UPDATE 串行化，但每条都是微秒级完成，不需要「先 SELECT 再判断」。

### 5 个 worker 抢同一个任务

```python
def try_claim(task_id):
    cursor = conn.execute(
        "UPDATE tasks SET status='running', worker=? "
        "WHERE id=? AND status='ready'",
        (my_name, task_id)
    )
    return cursor.rowcount == 1   # 1 = 抢到，0 = 没抢到
```

时间线：

```
t=0    Worker A、B、C、D、E 同时发 UPDATE t1
       SQLite 写锁串行处理：
       A → status 是 'ready'，匹配 → 改成 'running'，worker='A' → rowcount=1
       B → status 已经是 'running' → 不匹配 → 不改 → rowcount=0
       C/D/E 同上 → rowcount=0
t=ε    A 抢到，开干
       B/C/D/E 立刻去找下一个任务（t2、t3...）
```

赢家 1 个，输家 4 个，但没有任何输家在「等待」。输家立刻知道结果，转身去抢下一条。数据库自己保证原子性，不需要任何外部锁服务。

### 等待型 vs CAS 型

等待型（用锁）：

```
Worker A: 取锁 → 持有 → 释放
Worker B: 等待............ → 取锁 → 持有 → 释放
Worker C: 等待............................. → 取锁 ...
```

队列越长越后面越惨，任何一个 worker 卡住后面全堵。

CAS 型：

```
Worker A: try claim t1 → 成功 → 干活
Worker B: try claim t1 → 失败 → try claim t2 → 成功 → 干活
Worker C: try claim t1 → 失败 → try claim t2 → 失败 → try claim t3 → 成功
```

没有「等待」，只有「试一下，不行换下一个」。输家立刻有完整控制权决定下一步——找别的任务？退出？sleep 后再扫？由业务说了算，不是被锁强制定住。

### CAS 的另一面：乐观锁

同样的写法可以防丢更新。比如两个人同时编辑同一篇文章：

```sql
-- 朴素写法：后写的覆盖先写的
UPDATE articles SET body=? WHERE id=42

-- CAS 写法：带上"我看到的版本号"
UPDATE articles SET body=?, version=version+1
WHERE id=42 AND version=?
```

同事先你 1 秒保存，version 从 3 变 4，你的 UPDATE 用 `version=3` 匹配不上、rowcount=0、保存失败，前端弹「内容已被修改，请刷新」，而不是悄无声息覆盖掉别人的改动。

### Hermes kanban 的实现

```sql
UPDATE tasks
SET claim_lock = ?,           -- 我的 worker 标识
    claim_expires = ?,        -- 15 分钟后过期
    status = 'running'
WHERE id = ?
  AND claim_lock IS NULL      -- ← CAS 守卫
```

- 抢到：rowcount=1，执行任务。
- 没抢到：rowcount=0，调度器跳过看下一条。

完全没有 retry loop，没有等待。10 个 worker 同时启动时，SQLite 用 WAL 写锁把这 10 个 UPDATE 串行化（每个微秒级），9 个 rowcount=0 直接去看下一个任务。整个调度像「快速试错」，不是「排队取号」。

再加上每个看板用独立 db 文件，争锁的范围被物理切分掉——board A 的 worker 和 board B 的 worker 根本不互相影响。

## 总结：它怎么避免死锁 / 队列堆积

| 风险 | 它的反制 |
|---|---|
| 两个 DEFERRED 事务升级写锁 → 死锁不可重试 | 全部用 `BEGIN IMMEDIATE`，开局争锁 |
| SQLite 内置 busy handler 的确定性 sleep → convoy 卡顿 | 短 `timeout=1s` + 应用层随机抖动 20–150ms |
| 进程内多线程争 cursor | 单连接 + `threading.Lock` 串行化 |
| 跨进程 worker 抢任务 | CAS update，输家 `rowcount=0` 跳过，永不重试 |
| 跨 board 串行化 | 物理切分：每 board 一个 db 文件 |
| WAL 涨到失控 | 每 50 次写 + close 时 `PRAGMA wal_checkpoint(PASSIVE)` |
| NFS/SMB 上 WAL 不可用 | 自动降级 `journal_mode=DELETE`，按 db 去重 warn |
| 并发 migration ALTER 撞车 | 捕获 "duplicate column name" 吞掉 |

最值得抄的两个核心思想：

1. 永远用 `BEGIN IMMEDIATE` + 短 timeout + 应用层带 jitter 的重试——这是 SQLite 多 writer 场景的最佳实践，比调高 `busy_timeout` 健壮得多。
2. 跨进程协调用 CAS 而不是锁——`UPDATE ... WHERE status_was_expected`，让输家根本不进入「等待」状态。
